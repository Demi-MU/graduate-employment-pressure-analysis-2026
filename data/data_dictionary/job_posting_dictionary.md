# 岗位样本数据字典：Job Posting Dictionary

## 一、文档目的

本数据字典用于规范本项目中的岗位样本数据结构。岗位样本数据是计算“硕士岗位吸纳比”“行业硕士吸纳比”“城市硕士吸纳比”和“行业—城市硕士吸纳比”的核心数据基础。为了保证后续分析结果具有一致性、可解释性和可复现性，所有岗位样本在进入正式分析前，都应按照本文档规定的字段、口径和清洗规则进行整理。

本项目中的岗位样本主要来自公开招聘平台、公共招聘服务平台、国家大学生就业服务平台、国企校招平台、事业单位招聘公告、高校与科研机构招聘公告，以及部分可公开访问的企业校招页面。由于不同来源的岗位信息格式不一致，原始字段往往存在缺失、混乱或表达不统一的问题，因此需要在原始数据基础上进行标准化处理。

岗位样本数据的核心任务不是复刻完整招聘市场，而是在明确样本范围的前提下，观察不同岗位对学历、经验、行业和城市的要求，并进一步估算市场对硕士学历的吸纳能力。所有字段都应服务于这一研究目标。

## 二、数据表名称与存储位置

岗位样本原始数据建议存储在：

```text
data/raw/job_posting_samples/
```

经过初步清洗后的数据建议存储为：

```text
data/interim/cleaned_job_postings.csv
```

最终用于分析的数据建议存储为：

```text
data/processed/job_demand_panel.csv
```

其中，`cleaned_job_postings.csv` 主要保留经过基础清洗但尚未完全标准化的岗位信息；`job_demand_panel.csv` 则应作为正式分析表，字段命名、取值标准和样本有效性都应符合本数据字典要求。

## 四、样本识别字段

### 4.1 job_id

`job_id` 是岗位样本的唯一标识符。

字段类型：字符串。

建议生成规则为：

```text
source + collect_date + 序号
```

例如：

```text
mohrss_20260601_0001
ncss_20260601_0002
guopin_20260601_0003
```

如果原始平台提供岗位ID，可以保留原始ID并与数据源共同构成唯一键。若不同平台出现相同岗位，应通过岗位名称、公司名称、城市、学历要求和岗位链接进行去重。

### 4.2 collect_date

`collect_date` 表示岗位信息采集日期。

字段类型：日期。

格式建议为：

```text
YYYY-MM-DD
```

该字段非常重要，因为招聘岗位具有较强时间敏感性。不同时间窗口的岗位样本不能简单混合比较。例如，春招季岗位结构可能不同于秋招季，毕业季前后的岗位数量也可能发生明显变化。

### 4.3 source

`source` 表示岗位来源平台或来源渠道。

字段类型：字符串。

示例取值包括：

```text
中国公共招聘网
国家大学生就业服务平台
国聘
央企招聘平台
地方事业单位招聘公告
高校人才招聘网
企业校招官网
BOSS直聘公开页面
智联招聘公开页面
前程无忧公开页面
```

该字段用于追踪数据来源，并在后续分析中控制不同来源带来的样本偏差。

### 4.4 source_type

`source_type` 表示岗位来源类型。

字段类型：分类变量。

建议取值为：

```text
公共招聘平台
高校毕业生就业平台
国企央企招聘平台
事业单位招聘公告
高校科研机构招聘公告
企业校招官网
市场化招聘平台
其他
```

该字段有助于区分公共部门岗位、市场化岗位和高校毕业生专属岗位。不同来源类型不应直接混合解释，而应在需要时进行分组分析。

## 五、岗位基础字段

### 5.1 job_title

`job_title` 表示岗位名称。

字段类型：字符串。

示例取值包括：

```text
数据分析师
算法工程师
产品经理
管理培训生
财务专员
工艺工程师
高中数学教师
科研助理
事业单位工作人员
银行管培生
```

岗位名称应尽量保留原始文本，不建议过度修改。后续可以通过 `function_standard` 对岗位职能进行标准化。

### 5.2 company_name

`company_name` 表示招聘单位名称。

字段类型：字符串。

该字段用于识别用人单位，并支持后续进行去重、单位类型判断和企业性质分析。对于事业单位、高校、科研机构和政府相关单位，应尽量保留完整名称。

### 5.3 employment_type

`employment_type` 表示岗位用工类型。

字段类型：分类变量。

建议取值为：

```text
全职
实习
兼职
劳务派遣
合同制
编制内
编制外
不明确
```

本项目的主分析对象是全职应届岗位。实习岗位、兼职岗位和劳务派遣岗位可以保留，但在计算硕士岗位吸纳比时应根据研究口径决定是否排除。

### 5.4 company_type

`company_type` 表示招聘单位性质。

字段类型：分类变量。

建议取值为：

```text
民营企业
国有企业
央企
外资企业
合资企业
事业单位
政府机关
高校
科研机构
社会组织
其他
不明确
```

该字段用于分析硕士岗位需求是否集中在公共部门、国企、高校、科研机构或市场化企业中。由于不同平台不一定提供单位性质，该字段可以通过单位名称、平台来源和岗位描述进行人工或规则辅助判断。

## 六、地区字段

### 6.1 province

`province` 表示岗位所在省级行政区。

字段类型：字符串。

示例取值：

```text
北京
上海
广东
浙江
四川
湖北
江苏
陕西
重庆
天津
```

直辖市的 `province` 与 `city` 可以相同。

### 6.2 city

`city` 表示岗位所在城市。

字段类型：字符串。

城市名称应进行标准化，避免同一城市出现多种写法。例如：

```text
北京市 → 北京
上海市 → 上海
深圳市 → 深圳
成都市 → 成都
```

如果岗位为全国、多地可选或远程办公，应在 `city` 中标记为：

```text
全国
多地
远程
不明确
```

在正式计算城市硕士吸纳比时，建议排除“全国”“多地”“远程”和“不明确”样本，或单独设为一类。

### 6.3 district

`district` 表示岗位所在区县。

字段类型：字符串。

该字段不是第一阶段核心字段，但在分析一线城市岗位空间分布时可能有价值。例如北京海淀、朝阳，上海浦东、徐汇，深圳南山、福田，杭州余杭、西湖等。

如果原始数据未提供区县信息，可以为空。

### 7.1 industry_raw

`industry_raw` 表示原始数据中的行业名称。

字段类型：字符串。

该字段应尽量保留平台或公告中的原始行业表达，例如：

```text

电子技术/半导体/集成电路

银行
证券

新能源
医药生物
公共管理
```

原始行业字段不直接用于最终分析，而是作为行业标准化的依据。

### 7.2 industry_standard

`industry_standard` 表示标准化后的行业分类。

字段类型：分类变量。

建议第一阶段使用以下行业分类：

```text
互联网与软件
电子半导体
金融
教育科研
制造业
医药医疗
公共部门
消费零售
房地产与建筑
新能源与汽车
专业服务
其他
不明确
```

行业标准化规则应保存在：

```text
data/interim/cleaned_industry_mapping.csv
```

在正式报告中，应说明行业归并规则。行业分类不宜过细，否则会导致样本量不足；也不宜过粗，否则无法识别硕士岗位吸纳能力差异。

### 7.3 function_raw

`function_raw` 表示原始岗位职能或岗位类别。

字段类型：字符串。

如果平台提供岗位职能分类，应保留原始字段。例如：


### 7.4 function_standard

`function_standard` 表示标准化后的岗位职能。

字段类型：分类变量。

建议取值为：

```text
研发技术
算法与数据
产品

市场
销售
财务
人力资源
行政职能
教师教研
科研

供应链
管培生
公共管理
其他
不明确
```

该字段用于分析硕士学历需求到底集中在哪些职能中。例如，算法、科研、教师、医药研发和半导体研发可能更偏好硕士，而销售、运营和基础职能岗位可能更多以本科为主。

## 八、学历字段

学历字段是本项目最重要的数据字段。所有学历相关判断都必须严格区分“硬性要求”和“软性偏好”。

### 8.1 education_required_raw

`education_required_raw` 表示岗位原始学历要求。

字段类型：字符串。

示例取值包括：

```text
学历不限
大专
大专及以上
本科
本科及以上
硕士
硕士及以上
博士
博士研究生
研究生学历
硕士优先
研究生优先
985/211硕士优先
```

该字段应保留原始文本，不建议覆盖修改。后续所有标准化判断都应基于该字段和岗位描述共同完成。

### 8.2 education_level_standard

`education_level_standard` 表示标准化学历要求。

字段类型：分类变量。

建议取值为：

```text
学历不限
大专
大专及以上
本科
本科及以上
硕士
硕士及以上
博士
不明确
```

标准化规则如下：

如果原始学历要求为“本科”“统招本科”，则标记为“本科”。

如果原始学历要求为“本科及以上”，则标记为“本科及以上”。

如果原始学历要求为“硕士”“硕士研究生”，则标记为“硕士”。

如果原始学历要求为“硕士及以上”“研究生及以上”，则标记为“硕士及以上”。

如果原始学历要求为“博士”“博士研究生”，则标记为“博士”。

如果原始学历要求为“硕士优先”“研究生优先”，但基本要求仍为本科及以上，则 `education_level_standard` 应标记为“本科及以上”，同时在 `is_master_preferred` 中标记为 1。

如果无法判断学历要求，则标记为“不明确”。

### 8.3 is_bachelor_required

`is_bachelor_required` 表示岗位是否要求本科及以上学历。

字段类型：二元变量。

取值规则：

```text
1 = 岗位明确要求本科、本科及以上、硕士、硕士及以上或博士
0 = 岗位未要求本科及以上
```

该字段用于计算本科岗位主导指数。需要注意，硕士及以上和博士岗位也属于本科及以上门槛之上，因此应计入 `is_bachelor_required = 1`。

### 8.4 is_master_required

`is_master_required` 表示岗位是否硬性要求硕士及以上学历。

字段类型：二元变量。

取值规则：

```text
1 = 岗位明确要求硕士、硕士及以上、研究生及以上或博士
0 = 岗位未明确要求硕士及以上
```

示例：

```text
硕士 → 1
硕士及以上 → 1
研究生及以上 → 1
博士 → 1
本科及以上 → 0
硕士优先 → 0
研究生优先 → 0
985/211硕士优先 → 0
学历不限 → 0
```

这是计算“硕士硬性吸纳比”的核心字段。必须注意，“硕士优先”不是硬性要求，不能计入 `is_master_required = 1`。

### 8.5 is_master_preferred

`is_master_preferred` 表示岗位是否对硕士学历存在软性偏好。

字段类型：二元变量。

取值规则：

```text
1 = 岗位描述中出现硕士优先、研究生优先、硕博优先、985/211硕士优先等表述
0 = 未出现硕士偏好表述
```

示例：

```text
本科及以上，硕士优先 → is_master_required = 0, is_master_preferred = 1
本科及以上，研究生优先 → is_master_required = 0, is_master_preferred = 1
硕士及以上 → is_master_required = 1, is_master_preferred = 0
硕士及以上，博士优先 → is_master_required = 1, is_master_preferred = 0
```

在计算硕士偏好吸纳比时，可使用：

```text
is_master_required = 1 或 is_master_preferred = 1
```

## 九、应届与经验字段

### 9.1 is_fresh_graduate

`is_fresh_graduate` 表示岗位是否面向应届毕业生或接受应届毕业生。

字段类型：二元变量。

取值规则：

```text
1 = 岗位明确面向应届生、校招生、毕业生，或明确接受应届生
0 = 岗位不面向应届生，或无法判断为应届岗位
```

判定关键词包括：

```text
应届
校招
校园招聘
2026届
2025届
毕业生
管培生
储备干部
可接受应届生
```

需要注意，“经验不限”不必然等于应届岗位。若岗位仅写“经验不限”，但没有出现应届相关表述，可以根据研究口径将其标记为弱应届岗位，而不是直接标记为强应届岗位。

### 9.2 fresh_graduate_type

`fresh_graduate_type` 表示应届岗位类型。

字段类型：分类变量。

建议取值为：

```text
强应届
弱应届
非应届
不明确
```

判定规则如下：

如果岗位包含“应届”“校招”“校园招聘”“2026届”“毕业生”“管培生”等明确校招或应届关键词，则标记为“强应届”。

如果岗位仅包含“经验不限”“1年以内”“可接受应届生”等低经验要求表述，但没有明确校招属性，则标记为“弱应届”。

如果岗位明确要求两年以上工作经验，或明确要求社会招聘经验，则标记为“非应届”。

如果无法判断，则标记为“不明确”。

本项目在计算核心指标时，建议同时保留两个口径：

狭义应届口径：仅包含强应届岗位。

广义应届口径：包含强应届岗位和弱应届岗位。

### 9.3 experience_required_raw

`experience_required_raw` 表示原始经验要求。

字段类型：字符串。

示例取值包括：

```text
经验不限
应届毕业生
1年以内
1-3年
3-5年
5年以上
无需经验
有相关实习经历优先
```

该字段应保留原始文本，用于后续经验标准化。

### 9.4 experience_year_min

`experience_year_min` 表示岗位要求的最低工作经验年限。

字段类型：数值。

示例规则：

```text
经验不限 → 0
应届毕业生 → 0
1年以内 → 0
1-3年 → 1
3-5年 → 3
5年以上 → 5
```

如果无法判断，则为空。

### 9.5 experience_year_max

`experience_year_max` 表示岗位要求的最高工作经验年限。

字段类型：数值。

示例规则：

```text
经验不限 → 0
应届毕业生 → 0
1年以内 → 1
1-3年 → 3
3-5年 → 5
5年以上 → 空
```

如果无法判断，则为空。

经验字段用于判断岗位是否真正适合应届毕业生。如果岗位要求三年以上经验，即使学历要求较低，也不应计入应届岗位池。

## 十、薪资字段

薪资字段用于辅助分析岗位质量和机会成本，但不是第一阶段最核心字段。由于不同平台薪资表达不一致，薪资字段应尽量标准化。

### 10.1 salary_raw

`salary_raw` 表示原始薪资文本。

字段类型：字符串。

示例取值包括：

```text
8k-12k
10k-15k·13薪
15-25万/年
面议
5000-8000元/月
```

该字段应保留原始文本。

### 10.2 salary_min

`salary_min` 表示标准化后的薪资下限。

字段类型：数值。

建议统一为月薪人民币，单位为元/月。

示例：

```text
8k-12k → 8000
5000-8000元/月 → 5000
15-25万/年 → 12500
```

年薪换算为月薪时，可按12个月计算。若出现13薪、14薪，可在注释中保留，也可以统一转为年薪后再折算月薪。第一阶段建议采用简化方式，统一折算为月薪区间。

### 10.3 salary_max

`salary_max` 表示标准化后的薪资上限。

字段类型：数值。

建议统一为月薪人民币，单位为元/月。

示例：

```text
8k-12k → 12000
5000-8000元/月 → 8000
15-25万/年 → 20833
```

如果岗位薪资为“面议”，则 `salary_min` 和 `salary_max` 均为空。

### 10.4 salary_unit

`salary_unit` 表示原始薪资单位。

字段类型：分类变量。

建议取值为：

```text
元/月
千/月
万/年
面议
不明确
```

该字段用于回溯薪资换算规则。

## 十一、文本与技能字段

### 11.1 job_description

`job_description` 表示岗位描述文本。

字段类型：长文本。

该字段可以包括岗位职责、任职要求、专业要求、技能要求和其他说明。岗位描述是判断硕士优先、专业要求、应届属性和技能关键词的重要依据。

### 11.2 skill_keywords

`skill_keywords` 表示从岗位描述中提取的技能关键词。

字段类型：字符串或列表。

示例取值包括：

```text
Python, SQL, Excel, Tableau, Power BI, 机器学习, 深度学习, Java, C++, CAD, AutoCAD, SPSS, Stata, 论文写作, 实验设计
```

该字段用于后续扩展分析，即判断不同岗位是否更看重技能能力而非学历本身。第一阶段可以先手工或半自动提取高频关键词，不必进行复杂自然语言处理。

### 11.3 url

`url` 表示岗位来源链接。

字段类型：字符串。

该字段用于追溯岗位来源。对于招聘公告类数据，应保留公告页面链接；对于平台岗位，如果链接会失效，也应尽量保留原始页面地址或截图记录路径。

## 十二、样本有效性字段

### 12.1 valid_flag

`valid_flag` 表示岗位样本是否可用于正式分析。

字段类型：二元变量。

取值规则：

```text
1 = 有效样本
0 = 无效样本
```

有效样本应满足以下条件：

岗位名称明确。

招聘单位明确。

城市或工作地点明确，或可归入“全国/多地/远程”。

学历要求明确，或可以合理标记为“不明确”。

岗位不是明显重复样本。

岗位不是纯实习、兼职或无关岗位，除非研究口径明确包含。

岗位处于采集时间窗口内。

无效样本包括：

重复岗位。

岗位信息严重缺失。

岗位已过期且无法确认。

岗位与高校毕业生就业无关。

岗位为纯兼职、临时劳务、代理招聘且信息不完整。

无法判断工作地点、学历要求和岗位性质。

### 12.2 invalid_reason

`invalid_reason` 表示样本无效原因。

字段类型：字符串。

建议取值为：

```text
重复岗位
信息缺失
非应届相关岗位
非全职岗位
岗位过期
学历无法判断
地点无法判断
行业无法判断
无关样本
其他
```

如果 `valid_flag = 1`，该字段可以为空。

### 12.3 note

`note` 表示备注。

字段类型：字符串。

该字段用于记录特殊情况。例如：

```text
岗位为多城市招聘
岗位要求本科及以上但硕士优先
岗位来源公告包含多个岗位
薪资为年薪估算
单位性质需人工确认
```

## 十七、正式分析样本口径

在计算核心指标时，建议采用以下默认样本口径：

时间范围：明确采集日期所在的就业季或分析期。

岗位类型：优先使用全职岗位。

样本状态：仅使用 `valid_flag = 1` 的样本。

应届口径：默认使用狭义应届口径，即 `fresh_graduate_type = 强应届`。同时保留广义应届口径作为稳健性检验。

学历口径：默认排除 `education_level_standard = 不明确` 的样本。对于“硕士优先”岗位，不计入硕士硬性要求，但计入硕士偏好吸纳。

城市口径：城市分析中默认排除“全国”“多地”“远程”“不明确”样本，或单独归为特殊类别。

行业口径：行业分析中默认排除“不明确”行业样本，或单独归为“不明确”。

按照上述口径，核心指标计算方式为：

```text
硕士硬性吸纳比 =
valid_flag = 1 且 is_fresh_graduate = 1 且 is_master_required = 1 的岗位数
/
valid_flag = 1 且 is_fresh_graduate = 1 的岗位数
```

```text
硕士偏好吸纳比 =
valid_flag = 1 且 is_fresh_graduate = 1 且
(is_master_required = 1 或 is_master_preferred = 1) 的岗位数
/
valid_flag = 1 且 is_fresh_graduate = 1 的岗位数
```

## 十八、数据质量检查规则

岗位样本进入正式分析前，应进行以下检查。

第一，检查 `job_id` 是否唯一。如果存在重复ID，应重新生成或去重。

第二，检查核心字段缺失情况，包括 `job_title`、`company_name`、`city`、`industry_standard`、`education_level_standard`、`is_master_required`、`is_fresh_graduate` 和 `valid_flag`。

第三，检查学历字段逻辑是否一致。例如，如果 `education_level_standard = 硕士及以上`，则 `is_master_required` 必须为 1；如果 `education_level_standard = 本科及以上` 且仅写“硕士优先”，则 `is_master_required` 应为 0，`is_master_preferred` 应为 1。

第四，检查经验字段是否与应届字段冲突。例如，如果岗位要求 `experience_year_min >= 3`，则通常不应标记为强应届岗位，除非岗位描述中明确说明接受应届博士或特殊人才计划。

第五，检查薪资字段是否存在异常值。例如，月薪低于当地最低工资或高于明显不合理水平的样本，应回看原始薪资文本，判断是否是单位换算错误。

第六，检查行业与职能是否过度集中。如果某个行业或城市样本量过小，不应单独解释其硕士吸纳比，而应在报告中标注样本不足。

## 十九、局限性说明

岗位样本数据是观察招聘需求的重要窗口，但不能被视为全国就业市场的完整反映。不同平台的岗位发布习惯、行业覆盖范围、城市分布和企业类型都可能存在偏差。公共招聘平台可能更偏公共就业服务岗位，市场化招聘平台可能更偏企业白领岗位，国企央企平台则更偏稳定单位和校招岗位。

因此，本项目基于岗位样本计算出的硕士岗位吸纳比，应被理解为“样本范围内的观察指标”，而不是全国绝对比例。正式报告和视频表达中，应明确说明样本来源、时间窗口、城市范围、行业范围和岗位口径。

同时，招聘文本中的学历要求并不完全等同于企业真实录用标准。一些岗位写“本科及以上”，实际录用中可能偏好硕士；一些岗位写“硕士优先”，实际也可能录用优秀本科生。因此，本项目将“硬性要求”和“软性偏好”分开计算，以尽量减少解释误差。

## 二十、维护与更新规则

本数据字典应随着项目推进持续更新。如果新增数据来源、字段或清洗规则，应在本文档中补充说明，并记录更新时间。所有字段名称应尽量保持稳定，避免在不同Notebook和脚本中使用不同字段名表示同一含义。

如果后续项目扩展到更大规模岗位样本，可以进一步增加以下字段：

```text
major_required_raw
major_standard
degree_required
recruitment_batch
job_post_date
job_deadline
company_size
company_stage
remote_flag
campus_recruitment_flag
public_sector_flag
```

第一阶段不建议一次性加入过多字段。项目初期应优先保证学历、应届、行业、城市和岗位有效性字段的准确性，因为它们直接决定核心指标是否可信。
